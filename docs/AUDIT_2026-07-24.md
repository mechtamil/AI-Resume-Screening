# RecruitOS Deep End-to-End Codebase Audit

**Audit date:** 24-Jul-2026  
**Source:** `AI Recruitment Assistant(2).zip`  
**Audit mode:** Full repository/code/data/documentation review. No source files in the uploaded ZIP were modified.

## 1. Scope and audit coverage

The original uploaded archive contains **540 files** and **238 directory entries**.

- **330 files** are under `.git/`; all were inventoried and the repository was inspected using Git history, index, refs, object validation, and `git fsck`.
- **210 files** are project/application files outside `.git/`.
- **103 Python source files** were read in full and AST-parsed; all 103 are syntactically valid.
- **80 `.pyc` files** are generated bytecode caches. They were inventoried as binary artifacts; editable `.py` files are treated as source authority.
- **9 Excel workbooks** were opened and inspected sheet-by-sheet, including schemas and populated rows.
- **3 PDFs** were opened; page count, metadata, and extractable text were inspected.
- **1 SQLite database** was opened; schema, tables, and stored records were inspected.
- Markdown, TXT, JSON, TOML, log, config, and Git text metadata were read.

Every original ZIP file is listed in the companion manifest with its exact path, size, SHA-256, and review classification.

> Binary Git object files and `.pyc` files are not meaningful editable source text. They were inventoried and validated structurally rather than treated as source code.

## 2. Executive conclusion

RecruitOS has a useful backend foundation, especially the centralized repository layer, but the current project is **not yet an end-to-end working recruitment product**.

The audit found three unsynchronized realities:

1. **Git HEAD** — `c94e34e Sprint 5.6.0: Integrated master data into Resume Parser`
2. **The local working tree captured in the ZIP**
3. **The intended architecture discussed during development**

This mismatch is the main reason future development can stop unexpectedly.

## 3. P0 critical findings

### P0-1 — Git HEAD is broken on a clean clone

Committed `parser/resume_parser.py` imports:

`parser.extractors.skill_extractor`

But HEAD tracks only:

`parser/extractors/skills_extractor.py`

The plural tracked file is **0 bytes**.

The ZIP working tree contains the real singular `parser/extractors/skill_extractor.py`, but it is **untracked**, while the plural file is deleted locally.

A clean `git archive HEAD` was executed during the audit. Importing `parser.resume_parser` fails with:

`ModuleNotFoundError: No module named 'parser.extractors.skill_extractor'`

**Impact:** Sprint 5.6.0 is committed by name but cannot be reproduced from Git alone.

### P0-2 — Required central configuration workbook is untracked

The new repositories depend on:

`Master_Data/RecruitOS_Configuration.xlsx`

The file exists in the ZIP but is untracked.

A clean clone therefore has no workbook until the generator is run manually.

`python -m tools.create_configuration_workbook` works, but `MasterRepository` tells the user to run the direct form `python tools/create_configuration_workbook.py`, which previously failed in the user's environment because project imports were not resolved.

### P0-3 — Candidate name contract is broken

`PersonalExtractor.extract()` returns `full_name`.

`ResumeParser.parse()` reads `personal.get("name", "")`.

Actual execution on all supplied CVs produced blank candidate names.

### P0-4 — JD parsing is materially wrong

On `JD/sample_jd.txt`, the parser starts collecting at “Mandatory Skills” and never stops at later headers.

It produced **15 mandatory skills**, including preferred skills, `Education`, `BE Mechanical`, `Responsibilities`, and responsibility sentences.

Preferred skills, education, responsibilities, client, and location remained unparsed.

### P0-5 — Education extraction creates false positives

The central Education sheet includes aliases `BE` and `ME`.

The extractor scans the full resume case-insensitively. On the supplied Ajay resume, ordinary word “me” triggered `Master of Engineering`.

This was reproduced during the audit.

### P0-6 — UI does not invoke the backend

Static dependency tracing from `app.py` reaches only the Home/UI upload layer and `UploadService`.

It does not reach `ProcessingService`, parsers, matching, scoring, recommendation, or Results.

The Analyze button only simulates progress and states that the parser/matching engine will be connected later.

### P0-7 — Sensitive files are Git-tracked

Git tracks:
- 3 PDF CV/resume files with real-looking personal/contact data,
- `database/recruitos.db` containing candidate personal/employment/compensation data,
- application log data.

This is a privacy/data-governance risk, especially if repository visibility changes.

### Modern repository path

This layer is the strongest part of the current codebase.

### Resume backend path

`DocumentManager`
→ `ExtractionService`
→ `ResumeParser`
→ Personal/Skill/Education/Certification extractors

This exists, but has correctness/data issues.

### JD path

`DocumentManager`
→ `JDParser`

No master-data integration yet.

### Matching path

Two competing designs exist:

1. `services/matching_engine.py` — old monolithic engine
2. `services/matching/*.py` — modular matchers

No production orchestrator connects the modular matchers.

`services/matching/score_calculator.py` is 0 bytes.

No KeywordMatcher exists.

### UI path

`app.py`
→ `ui/home.py`
→ `ui/resume_screening.py`
→ `UploadService`

No processing/matching/results call path exists.

## 6. Model audit

### `JD/jd_model.py`
Active model; good base contract. Keep.

### `models/job_description.py`
Second incompatible JobDescription model; no active imports. Legacy duplicate.

### `models/candidate.py`
Useful base model, but:
- no persistence ID mapping,
- no LinkedIn/GitHub/website fields despite extraction,
- no DB-specific fields such as current company/notice/CTC,
- designation/tools/projects/companies/soft skills remain unpopulated.

### `models/match_result.py`
Modern score fields exist, but:
- `is_shortlisted()` hardcodes `>=70`,
- default recommendation hardcoded as `Review`,
- `certification_match` not declared,
- `skill_match_percentage` not declared.

Legacy matching code dynamically writes missing attributes.

## 7. Matching/scoring audit

### Legacy `services/matching_engine.py`
- writes undeclared `skill_match_percentage`,
- writes undeclared `certification_match`,
- changes `remarks` from declared list to string,
- hardcodes thresholds 85/70/50,
- uses temporary score bonuses,
- ignores ScoringRepository and RecommendationRepository.

Status: **do not extend as-is**.

### Modular matchers
- SkillMatcher: functional mandatory-skill percentage, loses canonical casing in result sets.
- ExperienceMatcher: functional minimum-experience scoring, ignores maximum/range semantics.
- EducationMatcher: exact canonical string intersection only.
- CertificationMatcher: functional percentage but writes undeclared `certification_match`.

Missing:
- KeywordMatcher,
- preferred-skill logic,
- ScoreCalculator,
- orchestrator,
- candidate ranking.

### ScoringRepository
Implemented; current active weights total 100.

### RecommendationRepository
Implemented, but ranges `0–59`, `60–74`, `75–89`, `90–100` leave decimal gaps. Scores `59.5`, `74.5`, and `89.5` return no recommendation.

### Legacy workbooks

`skills_master.xlsx` contains 5 skills including Java, Docker, Arbortext; only 2 were migrated centrally.

`certification_master.xlsx` contains a populated certification taxonomy; central Certifications contains zero rows.

`scoring_master.xlsx` duplicates the central scoring concept.

Other separate master workbooks are empty shells.

### Generator risk

`tools/create_configuration_workbook.py` unconditionally saves over the central workbook and inserts sample/business values.

There is no backup, overwrite guard, or migration protection.

`tools/create_skill_repository.py` still generates the legacy `skills_master.xlsx` architecture.

## 9. Parser/extractor audit

### PDF/DOCX readers
Catch all errors and return empty strings.

Missing/corrupt files therefore look like empty successful documents.

TXT reader raises normally, so behavior is inconsistent.

### DocumentManager
Good standardized dictionary contract, but cannot distinguish read failure from valid empty document because readers swallow errors.

### ExtractionService
Basic cleaning works.

Paragraph metadata is unreliable because blank lines are removed before paragraph counting.

Detected sections are not effectively consumed by current parsers.

### Personal extraction
Duplicate old extractors coexist with PersonalExtractor.

Critical mismatch: `full_name` vs `name`.

Hardcoded location list conflicts with planned Locations repository.

LinkedIn/GitHub/website values are extracted and discarded.

### Skill extraction
Correct configuration-driven direction, but:
- central list is too small,
- punctuation-heavy skills may be fragile with `\b` patterns,
- repository loaded at import time,
- singular module is untracked.

### Education extraction
Configuration-driven but unsafe for short/common aliases.

### Certification extraction
Architecture is reasonable, but central sheet is empty, so real extraction is effectively disabled.

## 19. Repository-layer caveats

### SkillRepository
No explicit duplicate-standard-name rejection and no ambiguous-synonym collision protection; later entries can silently overwrite earlier mappings.

### EducationRepository
Ambiguous aliases across different degrees are detected, but duplicate standard degree names can overwrite silently.

### CertificationRepository
Strongest of the three: duplicate standards and ambiguous aliases are validated.

### MasterRepository
Validates required sheets only, not schemas/data quality; caches workbook globally; error guidance contains wrong direct-run command.
